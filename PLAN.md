# Plan: Multi-Account Login With PostgreSQL Storage

## 1. Current Architecture Analysis

### Repository shape

`lite-chat` is currently a deliberately small Fastify application:

- `server.js`: one ESM Fastify server.
- `public/index.html`: one static HTML file containing markup, CSS, and vanilla JavaScript.
- `package.json`: minimal runtime dependencies for Fastify, static serving, multipart upload, and CORS.
- No database, no backend session state, no build step, no frontend framework.

### Backend behavior

The Fastify server currently does four jobs:

- Serves `public/` at `/`.
- Serves uploaded images from `uploads/` at `/uploads/`.
- Accepts image uploads at `POST /api/upload`.
- Proxies LLM calls:
  - `GET /api/models` tries OpenAI-compatible `/v1/models`, then Ollama `/api/tags`.
  - `POST /api/chat` streams OpenAI-compatible `/v1/chat/completions` through to the browser as SSE.

The backend currently trusts client-provided headers:

- `x-base-url`: selected LLM backend URL.
- `x-api-key`: selected LLM API key.

This means the browser owns provider configuration today. The server does not know who the user is and does not isolate any data by user.

### Frontend behavior

The frontend is a single vanilla JS app in `public/index.html`.

State is stored in `localStorage`:

- `sessions`: object keyed by generated client IDs, containing `{ title, messages, sysPrompt }`.
- `sessionIds`: ordered list of conversation IDs.
- `curId`: active conversation ID.
- `baseUrl`: LLM backend URL.
- `apiKey`: LLM API key.
- `lastModel`: selected model.
- `globalSysPrompt`: default system prompt.

The UI includes:

- Sidebar with local conversation list.
- Settings panel with Base URL, API key, and model selector.
- Main chat surface.
- System prompt modal.
- Image upload and paste support.

### Constraints implied by the existing design

- Keep a no-build frontend unless there is a strong reason to change it.
- Keep Fastify and vanilla JS.
- Keep streaming chat behavior unchanged from the user's perspective.
- Avoid introducing a large ORM or full authentication framework.
- Preserve the current "works locally with Ollama" path while adding authenticated multi-account support.

## 2. Target Architecture

### Goal

Add multi-account login support so multiple users can use the same lite-chat instance while keeping their identities, LLM credentials, conversations, and messages isolated in PostgreSQL.

PostgreSQL will run in Docker through Docker Compose. The Node app can run either on the host machine or in Docker later; this plan only requires PostgreSQL in Docker.

### High-level design

The app remains a lightweight Fastify + vanilla JS application:

- Fastify serves static files and API routes.
- PostgreSQL stores users, account/provider settings, sessions, conversations, and messages.
- `pg` provides a small database connection pool.
- `@fastify/jwt` signs and verifies auth tokens.
- `bcrypt` hashes user passwords.
- The browser stores only the JWT and minimal UI state, not API keys or conversation history.

### Account model

Use "user" for the person who logs in. Use "account" for a per-user LLM backend configuration.

Example:

- User: `alice@example.com`
- Accounts:
  - `Local Ollama`: `http://localhost:11434`, no API key
  - `OpenAI Compatible Gateway`: `https://gateway.example.com`, API key

Each user can create multiple LLM accounts and switch between them in the UI. Conversations belong to one user and may optionally be associated with the selected LLM account used for that conversation.

### Request flow

Authentication:

1. User registers or logs in.
2. Fastify validates credentials.
3. Fastify returns a JWT signed by `@fastify/jwt`.
4. Browser sends `Authorization: Bearer <token>` on authenticated API calls.

Chat:

1. Browser selects an LLM account.
2. Browser sends conversation/message request without API key headers.
3. Fastify loads the selected account from PostgreSQL, scoped to `request.user.id`.
4. Fastify uses the stored `base_url` and decrypted/API-key value to call the upstream LLM API.
5. Fastify streams the upstream response to the browser.
6. Fastify persists user and assistant messages in PostgreSQL.

## 3. Database Schema Design

Use plain SQL migrations stored in a new `migrations/` directory. Avoid an ORM.

Use UUID primary keys generated by PostgreSQL with `gen_random_uuid()` from `pgcrypto`.

### Extensions

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

### `users`

Stores login identities.

```sql
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  display_name TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_login_at TIMESTAMPTZ
);
```

Notes:

- Store normalized lowercase email.
- Never store raw passwords.
- `password_hash` is produced by `bcrypt`.

### `accounts`

Stores each user's LLM backend configurations.

```sql
CREATE TABLE accounts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  base_url TEXT NOT NULL,
  api_key_ciphertext TEXT,
  provider_type TEXT NOT NULL DEFAULT 'openai_compatible',
  default_model TEXT,
  is_default BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT accounts_name_per_user UNIQUE (user_id, name)
);

CREATE INDEX accounts_user_id_idx ON accounts(user_id);
```

Notes:

- `api_key_ciphertext` should be encrypted before storage using an application secret, not merely hidden.
- Keep `provider_type` simple initially: `openai_compatible` and optionally `ollama`.
- Ensure only one default account per user in application logic. A partial unique index can enforce this:

```sql
CREATE UNIQUE INDEX accounts_one_default_per_user_idx
ON accounts(user_id)
WHERE is_default = true;
```

### `sessions`

Tracks active/revocable login sessions. Even with JWTs, this gives the app a way to log out a specific device and expire tokens server-side.

```sql
CREATE TABLE sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token_jti UUID NOT NULL UNIQUE,
  user_agent TEXT,
  ip_address INET,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at TIMESTAMPTZ NOT NULL,
  revoked_at TIMESTAMPTZ
);

CREATE INDEX sessions_user_id_idx ON sessions(user_id);
CREATE INDEX sessions_token_jti_idx ON sessions(token_jti);
```

Notes:

- JWT payload includes `sub` as user ID and `jti` as session token ID.
- Middleware verifies both the JWT signature and that `jti` exists, is not revoked, and is not expired.

### `conversations`

Stores chat threads.

```sql
CREATE TABLE conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  account_id UUID REFERENCES accounts(id) ON DELETE SET NULL,
  title TEXT NOT NULL DEFAULT '新对话',
  system_prompt TEXT,
  model TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at TIMESTAMPTZ
);

CREATE INDEX conversations_user_id_updated_at_idx
ON conversations(user_id, updated_at DESC);
```

Notes:

- Every conversation query must include `user_id = request.user.id`.
- `account_id` is nullable so conversations survive if an LLM account is deleted.
- Keep `system_prompt` per conversation, matching current behavior.

### `messages`

Stores chat messages.

```sql
CREATE TABLE messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('system', 'user', 'assistant', 'tool')),
  content JSONB NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  token_count INTEGER,
  error TEXT
);

CREATE INDEX messages_conversation_created_at_idx
ON messages(conversation_id, created_at ASC);

CREATE INDEX messages_user_id_idx ON messages(user_id);
```

Notes:

- Use `JSONB` for `content` because current messages can be plain text or OpenAI multimodal arrays.
- Store assistant errors as failed assistant messages only if the UI should display them later. Otherwise, omit failed responses.
- `user_id` duplicates the conversation ownership for simpler authorization checks and cleanup.

### Optional `uploads`

The current app stores files on disk and returns `/uploads/<filename>`. That can remain unchanged in the first pass.

Later, add:

```sql
CREATE TABLE uploads (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  filename TEXT NOT NULL,
  original_filename TEXT,
  mime_type TEXT,
  byte_size INTEGER,
  url_path TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

This is useful for ownership checks and cleanup, but it is not required for the initial auth/database migration.

Add `docker-compose.yml` at the project root:

```yaml
services:

    image: postgres:16-alpine
    container_name: lite-chat-postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: lite_chat
      POSTGRES_USER: lite_chat
      POSTGRES_PASSWORD: lite_chat_dev_password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U lite_chat -d lite_chat"]
      interval: 5s
      timeout: 5s
      retries: 10

volumes:
  postgres_data:
```

Add `.env.example`:

```env
PORT=3131
DATABASE_URL=postgres://lite_chat:lite_chat_dev_password@localhost:5432/lite_chat
JWT_SECRET=change-me-dev-secret

```

Notes:

- Do not commit a real `.env`.
- Compose only runs PostgreSQL for now to preserve the lightweight local Node workflow.
- A later phase can add an app service if containerized deployment is desired.

## 5. Backend Changes Needed

### Dependencies

Add:

- `@fastify/jwt`: Fastify-native JWT signing and verification.
- `pg`: PostgreSQL connection pool and query client.
- `bcrypt`: password hashing.
- `dotenv`: optional local `.env` loading.

Keep avoiding:

- Heavy ORMs.
- Full auth frameworks.
- Frontend build tooling.

### Suggested backend file layout

Keep `server.js` as the application entry point, but split support code into small modules:

```text
server.js
lib/
  db.js
  auth.js
  crypto.js
  migrations.js
routes/
  auth.js
  accounts.js
  conversations.js
  chat.js
  upload.js
migrations/
  001_initial_auth_and_chat.sql
scripts/
  migrate.js
```

This keeps the app lightweight while avoiding a single large `server.js`.

### DB connection pool

`lib/db.js`:

- Create one `pg.Pool` from `DATABASE_URL`.
- Export `query(text, params)` and `withTransaction(fn)`.
- Add shutdown handling to call `pool.end()` when the server exits.

### Password hashing

Use `bcrypt`:

- Hash with a cost factor around `12` for normal deployment.
- Keep the cost configurable for tests if needed.
- Use `bcrypt.compare()` during login.

Rationale:

- Battle-tested password hashing.
- Salt included in the hash.
- Simple API and no new service dependency.

### JWT auth

Use `@fastify/jwt`.

JWT payload:

```json
{
  "sub": "<user_id>",
  "email": "<email>",
  "jti": "<session_token_jti>"
}
```

Middleware:

- Register JWT plugin with `JWT_SECRET`.
- Add `app.decorate('authenticate', async function (request, reply) { ... })`.
- Verify token signature.
- Load `sessions` row by `jti`.
- Reject if missing, expired, or revoked.
- Attach `request.user = { id, email, sessionId, jti }`.

Rationale for JWT plus session table:

- Browser API calls are simple with `Authorization: Bearer`.
- Tokens work cleanly with SSE/proxy fetch requests.
- The session table restores revocation/logout capability that pure stateless JWTs lack.
- Still simpler than introducing server-side cookie session middleware.

Token storage choice:

- Minimum-change option: store access token in `localStorage`.
- Safer option: use an HttpOnly cookie for the JWT.

Recommended first implementation:

- Use `Authorization: Bearer` from `localStorage` for simplicity and because the current app already uses browser JS-owned state.
- Keep token TTL modest, for example 7 days.
- Add clear XSS discipline: escape rendered user text and avoid injecting untrusted HTML outside Markdown rendering.

### Auth routes

`POST /api/auth/register`

Request:

```json
{ "email": "alice@example.com", "password": "secret", "displayName": "Alice" }
```

Behavior:

- Validate email format and password length.
- Normalize email.
- Hash password.
- Insert user.
- Create default account using `DEFAULT_BASE_URL` if desired.
- Create session row.
- Return JWT and user profile.

`POST /api/auth/login`

- Validate credentials.
- Update `last_login_at`.
- Create session row.
- Return JWT and user profile.

`POST /api/auth/logout`

- Requires auth.
- Mark current session `revoked_at = now()`.

`GET /api/auth/me`

- Requires auth.
- Returns current user and basic settings.

### Account routes

All require auth and scope by `user_id`.

`GET /api/accounts`

- List current user's LLM accounts.
- Do not return raw API keys.
- Return `hasApiKey: true/false`.

`POST /api/accounts`

- Create account.
- Validate `base_url`.
- Encrypt API key if provided.

`PATCH /api/accounts/:id`

- Update name, base URL, API key, default model, default flag.
- If API key is omitted, keep existing key.
- If API key is empty string, clear existing key.

`DELETE /api/accounts/:id`

- Delete only if owned by current user.
- Existing conversations keep `account_id = NULL` because of `ON DELETE SET NULL`.

`GET /api/accounts/:id/models`

- Load account by `id` and `user_id`.
- Decrypt API key.
- Try OpenAI-compatible `/v1/models`, then Ollama `/api/tags`.
- Return models.

### Conversation routes

All require auth and scope by `user_id`.

`GET /api/conversations`

- Return ordered list: `id`, `title`, `account_id`, `model`, `updated_at`.
- Do not return all messages in the list endpoint.

`POST /api/conversations`

- Create a new conversation with title, system prompt, account, and model.

`GET /api/conversations/:id`

- Return conversation metadata and messages ordered by `created_at`.

`PATCH /api/conversations/:id`

- Update title, system prompt, account, model, archived state.

`DELETE /api/conversations/:id`

- Delete or archive. For parity with current behavior, hard delete is acceptable initially.

### Chat route

Replace current `POST /api/chat` behavior with an authenticated DB-backed route.

Request:

```json
{
  "conversationId": "<uuid>",
  "accountId": "<uuid>",
  "model": "gpt-4.1-mini",
  "message": {
    "role": "user",
    "content": "hello"
  },
  "stream": true
}
```

Behavior:

1. Authenticate user.
2. Verify conversation belongs to user.
3. Verify account belongs to user.
4. Insert user message.
5. Load conversation history from `messages`.
6. Prepend system prompt if present.
7. Call upstream LLM with account base URL/API key.
8. Stream SSE chunks to browser.
9. Accumulate assistant content server-side.
10. Insert assistant message after stream completes.
11. Update conversation `updated_at`, `title`, `model`, and `account_id` if needed.

Failure handling:

- If upstream request fails before streaming, return JSON error and keep the user message.
- If streaming fails mid-response, end stream and either:
  - store partial assistant content with `error`, or
  - omit assistant content and let UI show a transient error.

Recommended first pass:

- Store partial assistant content with `error` only if any content was received.
- Otherwise do not insert assistant message.

### Legacy compatibility routes

For migration safety, keep unauthenticated behavior temporarily behind a flag:

- If `AUTH_REQUIRED=false`, allow current header-based `/api/models` and `/api/chat`.
- Default new behavior should require auth once the login UI is ready.

This lets development proceed without breaking the current UI on the first backend commit.

### API helper

Create a small client helper in the inline script:

```js
async function api(path, options = {}) {
  return fetch(path, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(authToken ? { Authorization: `Bearer ${authToken}` } : {}),
      ...(options.headers || {}),
    },
  })
}
```

Do not use this helper for `FormData` upload unless it conditionally avoids setting `Content-Type`.

- Default model

Change `sendMessage()`:

### Per-user isolation

The frontend does not enforce isolation. It only uses authenticated APIs.

Isolation must be enforced by backend route queries:

- Every `accounts` query includes `user_id`.
- Every `conversations` query includes `user_id`.
- Every message query joins or checks the owning conversation.

Initial migration:

- Require auth for `POST /api/upload`.

- Use the returned URL in message content as currently done.

Later:

- Store upload metadata in DB.

## 7. Migration Strategy

The safest path is additive and reversible.

### Step 1: Add DB without changing current behavior

- Add PostgreSQL Compose file.
- Add DB pool and migration script.
- Add schema.
- Add health check route like `GET /api/health/db`.
- Existing chat UI still uses localStorage and header-provided credentials.

### Step 2: Add auth API behind unused routes

- Add register/login/logout/me.
- Add session middleware.
- Add tests/manual curl checks.
- Existing UI still works.

### Step 3: Add account API

- Add account CRUD.
- Keep existing `/api/models` header behavior.
- Add authenticated account-scoped model route.

### Step 4: Add conversation API

- Add CRUD for conversations and messages.
- Keep current localStorage UI unchanged until endpoints are stable.

### Step 5: Switch frontend after backend is ready

- Add login/register UI.
- Load accounts and conversations from API.
- Use authenticated account-scoped chat.

### Step 6: LocalStorage import

Offer an optional one-time import:

- If user logs in and localStorage contains old `sessions`, show "Import local conversations".
- Create DB conversations/messages for the current logged-in user.
- Use current `baseUrl`, `apiKey`, and `lastModel` to create an LLM account if the user chooses.
- After successful import, mark `localStorage.migratedToDb = true`.

This prevents data loss while avoiding automatic upload of old local data without user intent.

### Step 7: Remove or gate legacy behavior

After DB-backed UI works:

- Make auth required by default.
- Keep legacy mode only behind `AUTH_REQUIRED=false` for local development.
- Stop accepting browser-supplied `x-api-key` when authenticated mode is enabled.

## 9. Tech Choices and Rationale

### PostgreSQL

PostgreSQL is a good fit because conversations, users, sessions, and accounts are relational, while message content benefits from `JSONB`. Running it in Docker keeps local setup reproducible without making the Node app itself heavier.

### `pg` / node-postgres

Use `pg` directly instead of an ORM.

Rationale:

- Small dependency.
- Clear SQL.
- Easy connection pooling.
- Fits the existing lightweight philosophy.
- Avoids schema/model layers that would be disproportionate for this app.

### SQL migrations

Use plain `.sql` migrations plus a tiny Node runner.

Rationale:

- Transparent schema history.
- No heavy migration framework.
- Easy to run in Docker/local development.

### `@fastify/jwt`

Use `@fastify/jwt` for auth tokens.

Rationale:

- Fastify-native.
- Minimal API.
- Works well with `Authorization: Bearer`.
- Easy to combine with a `sessions` table for revocation.

### JWT plus session table

Pure JWTs are simple but hard to revoke. Server-side cookie sessions are revocable but require more browser/cookie/CORS decisions.

JWT plus a `sessions` table gives:

- Simple frontend fetch integration.
- Stateless token verification for shape/signature.
- Server-side logout and revocation through `jti`.
- Multiple active devices per user.

### `bcrypt`

Use `bcrypt` for password hashing.

Rationale:

- Mature and widely used.
- Per-password salt is built in.
- Cost factor can be tuned.
- Better than generic hashes like SHA-256 for passwords.

### API key encryption

Store API keys encrypted in the database using an app-level secret.

Rationale:

- API keys must not be stored in plaintext.
- The server needs to decrypt them to call upstream LLM APIs.
- Hashing is not suitable because the original key is needed.

Implementation option:

- Use Node's built-in `crypto` module with AES-256-GCM.
- Store ciphertext, IV, and auth tag in one encoded string.
- Require `API_KEY_ENCRYPTION_SECRET` at startup.

### Vanilla JS frontend

Keep the frontend as a single static page for now.

Rationale:

- Existing app is already functional and small.
- Login, account switching, and DB-backed conversations do not require a framework.
- Avoiding a build step keeps deployment simple.

## Security and Ownership Rules

- Never trust `user_id`, `account_id`, or `conversation_id` from the client without checking database ownership.
- Never return `password_hash` or raw API keys.
- Normalize emails before storing and comparing.
- Use parameterized SQL for every query.
- Use generic login failure messages.
- Require HTTPS in production if JWTs are stored in localStorage.
- Consider moving JWTs to HttpOnly cookies in a later hardening pass.
- Restrict CORS in production; current `origin: '*'` is acceptable only for local development.

- `POST /api/upload`

## Acceptance Criteria

The implementation is complete when:

- PostgreSQL starts with Docker Compose and migrations create the schema.
- Users can register, log in, refresh, and log out.
- Each user can create and switch between multiple LLM accounts.
- API keys are stored encrypted and are not sent back to the browser.
- Model loading uses the selected stored account.
- Conversations and messages persist in PostgreSQL.
- Users cannot access each other's accounts, conversations, messages, or sessions.
- Chat streaming still works in the existing UI style.
- Existing localStorage conversations can be imported without automatic data loss.
- The app remains Fastify + vanilla JS with no heavy frontend or ORM framework.
